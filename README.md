## What You Will Build

By the end of this lab, you will have a fully automated DevSecOps pipeline that does the following **without any human intervention**:

1. Detects outdated dependencies automatically
2. Opens a Pull Request for each update
3. Runs security scanning on the new version
4. Auto-merges if all checks pass
5. Deploys to production on Vercel

```
Developer pushes code
        ↓
Renovate checks dependencies
        ↓
Creates PR (if update found)
        ↓
GitHub Actions runs CI:
    - npm install
    - tests
    - Snyk security scan
        ↓
If green → auto-merge
        ↓
Merge to main
        ↓
GitHub Actions triggers CD
        ↓
Deploy to Vercel
        ↓
Live production URL updated
```

## Concepts

Before writing any code, understand what each tool does.

| Tool | What it does | When it runs |
|---|---|---|
| **Renovate** | Scans `package.json` and opens PRs for outdated dependencies | Every weekday (scheduled) |
| **GitHub Actions** | Runs automated workflows on events like push or pull request | On every PR and every push to main |
| **Snyk** | Scans `node_modules` for known security vulnerabilities | Inside the CI workflow on every PR |
| **Vercel** | Builds and hosts your Node.js app on a global edge network | After every successful merge to main |

**Why keep CI and CD in separate files?**  
CI runs on Pull Requests. CD runs on merges to main. They serve different purposes. Mixing them in one file makes the pipeline harder to debug and reason about.

**What is a security gate?**  
A step in the pipeline that blocks deployment if a security check fails. In this lab, Snyk is the security gate. If it finds a medium or higher vulnerability in a new dependency version, the PR is blocked and nothing reaches production.

## Step 1 — Set Up the Project

### 1.1 Create the app files

Create a new folder and add the following files.

**`package.json`**

```json
{
  "name": "devsecops-sample-app",
  "version": "1.0.0",
  "description": "Sample Node.js app for DevSecOps pipeline",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "build": "mkdir -p build && cp -r index.js package.json build/ && echo 'Build completed'",
    "test": "echo 'Running tests' && exit 0"
  },
  "keywords": ["devsecops", "node", "ci-cd"],
  "author": "Your Name",
  "license": "MIT",
  "dependencies": {
    "express": "4.17.0",
    "lodash": "^4.18.1",
    "minimatch": "^3.1.3"
  },
  "devDependencies": {
    "snyk": "^1.685.0"

> **Note for students:** Notice that `express` is pinned to `4.17.0` — an old version. Renovate will detect this and open a PR to update it. This is intentional.

**`index.js`**

const express = require('express');

const app = express();

  const obj = lodash.cloneDeep({
    message: 'Hello, DevSecOps!'

  const pattern = minimatch('.js', '.js');

  res.send(`${obj.message} (Pattern match: ${pattern})`);
});

module.exports = app;
```

**`vercel.json`**

      "src": "index.js",

### 1.2 Generate the lockfile

```bash
npm install
```

This creates `package-lock.json`. This file must be committed to your repo. GitHub Actions uses it to cache dependencies.

> **Why `npm ci` instead of `npm install` in CI?**  
> `npm ci` installs exactly what is in the lockfile with no modifications. It is faster and deterministic — every run produces the same result. `npm install` can update the lockfile which introduces unpredictability in CI.

### 1.3 Create a GitHub repository and push

```bash
git init
git add .
git commit -m "initial commit: devsecops sample app"
git remote add origin https://github.com/YOUR_USERNAME/YOUR_REPO.git
git push -u origin main
```

## Step 2 — Add GitHub Secrets

Your CI and CD workflows use API tokens that must never be hardcoded in files. GitHub Secrets store them securely and inject them as environment variables at runtime.

**How to add a secret:**  
Go to your repo → Settings → Secrets and variables → Actions → New repository secret

Add these four secrets:

| Secret name | How to get it |
|---|---|
| `SNYK_TOKEN` | Log in to [app.snyk.io](https://app.snyk.io) → Account Settings → Auth Token |
| `VERCEL_TOKEN` | Log in to Vercel → Settings → Tokens → Create token |
| `VERCEL_ORG_ID` | Run `vercel link` locally, then open `.vercel/project.json` — copy `orgId` |
| `VERCEL_PROJECT_ID` | Same file — copy `projectId` |

**Getting your Vercel IDs:**

```bash
npm install -g vercel
vercel login
vercel link
cat .vercel/project.json
```

The output looks like:

```json
{
  "orgId": "team_xxxxxxxxxxxx",
  "projectId": "prj_xxxxxxxxxxxx"
}
```

Copy those values into your GitHub Secrets.

> **Security reminder for students:** Secrets are never visible in logs. GitHub masks them automatically. Never `console.log` a secret or echo it in a shell command.

## Step 3 — Configure Renovate

### 3.1 Install the Renovate GitHub App

Go to [github.com/apps/renovate](https://github.com/apps/renovate) and click **Install**. Select your repository under "Only select repositories".

### 3.2 Create `renovate.json`

Create this file at the root of your project:

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "description": "Renovate config: auto-detects dependency updates, opens PRs, auto-merges if CI passes",

  "extends": ["config:base"],

  "timezone": "UTC",
  "schedule": ["every weekday"],

  "labels": ["dependencies", "renovate"],

  "prConcurrentLimit": 5,
  "prHourlyLimit": 2,

  "dependencyDashboard": true,
  "dependencyDashboardTitle": "Renovate Dependency Dashboard",

  "commitMessagePrefix": "chore(deps):",
  "commitMessageAction": "update",
  "commitMessageTopic": "{{depName}}",
  "commitMessageExtra": "from {{currentVersion}} to {{newVersion}}",

  "rangeStrategy": "bump",

  "packageRules": [
    {
      "description": "Auto-merge patch updates for devDependencies",
      "matchDepTypes": ["devDependencies"],
      "matchUpdateTypes": ["patch", "minor"],
      "automerge": true,
      "automergeType": "pr",
      "automergeStrategy": "squash"
    },
    {
      "description": "Auto-merge patch updates for production dependencies",
      "matchDepTypes": ["dependencies"],
      "matchUpdateTypes": ["patch"],
      "automerge": true,
      "automergeType": "pr",
      "automergeStrategy": "squash"
    },
    {
      "description": "Auto-merge major version bumps after CI passes",
      "matchUpdateTypes": ["major"],
      "automerge": true,
      "automergeType": "pr",
      "automergeStrategy": "squash",
      "labels": ["dependencies", "renovate", "major-update"],
      "prPriority": 5
    },
    {
      "description": "Security vulnerability fixes — highest priority",
      "matchDepTypes": ["dependencies", "devDependencies"],
      "matchUpdateTypes": ["patch"],
      "isVulnerabilityAlert": true,
      "automerge": true,
      "automergeType": "pr",
      "labels": ["dependencies", "renovate", "security"],
      "prPriority": 10
    },
    {
      "description": "GitHub Actions version updates — auto-merge",
      "matchManagers": ["github-actions"],
      "matchUpdateTypes": ["minor", "patch"],
      "automerge": true,
      "groupName": "GitHub Actions updates"
    }
  ],

  "vulnerabilityAlerts": {
    "enabled": true,
    "labels": ["security", "renovate"]
  },

  "npm": {
    "stabilityDays": 3
  }
}
```

**What the key settings mean:**

| Setting | Meaning |
|---|---|
| `schedule` | Renovate runs every weekday |
| `dependencyDashboard: true` | Creates a GitHub Issue listing all pending updates |
| `prConcurrentLimit: 5` | Max 5 open Renovate PRs at once |
| `automerge: true` | Auto-merges the PR if CI passes |
| `automergeStrategy: squash` | Squash-merges — keeps git history clean |
| `stabilityDays: 3` | Waits 3 days before updating (avoids broken new releases) |
| `isVulnerabilityAlert` | Security fixes skip the wait and get highest priority |

## Step 4 — CI Workflow

Create `.github/workflows/ci.yml`:

```yaml
name: CI — Test, Lint & Security Scan

on:
  pull_request:
    branches:
      - main
    types: [opened, synchronize, reopened]

  workflow_dispatch:

jobs:
  ci:
    name: Install, Test & Security Scan
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - name: Install Dependencies
        run: npm ci

      - name: Run Tests
        run: npm test

      - name: Install Snyk CLI
        run: npm install -g snyk

      - name: Authenticate Snyk
        run: snyk auth ${{ secrets.SNYK_TOKEN }}

      - name: Snyk Scan — Log Results
        run: |
          echo "========================================"
          echo "  Snyk Dependency Vulnerability Scan"
          echo "  PR: ${{ github.event.pull_request.title }}"
          echo "  Branch: ${{ github.head_ref }}"
          echo "  Commit: ${{ github.sha }}"
          echo "  Actor: ${{ github.actor }}"
          echo "========================================"
          snyk test --severity-threshold=medium --json | tee snyk-results.json || true
          snyk test --severity-threshold=medium 2>&1 | tee snyk-summary.txt

      - name: Upload Snyk Results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: snyk-scan-results
          path: |
            snyk-results.json
            snyk-summary.txt
          retention-days: 30

      - name: Fail if Medium+ Vulnerabilities Found
        run: snyk test --severity-threshold=medium

  auto-merge:
    name: Auto-Merge Renovate PR
    needs: ci
    runs-on: ubuntu-latest
    if: |
      github.event.pull_request.user.login == 'renovate[bot]' &&
      needs.ci.result == 'success'

    permissions:
      contents: write
      pull-requests: write

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Auto-Merge via GitHub CLI
        run: |
          echo "CI passed — merging PR #${{ github.event.pull_request.number }}"
          gh pr merge ${{ github.event.pull_request.number }} \
            --squash \
            --auto \
            --delete-branch \
            --subject "${{ github.event.pull_request.title }}" \
            --body "Auto-merged after passing CI:
          - npm install ✓
          - npm test ✓
          - Snyk scan ✓"
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**What each job does:**

The `ci` job runs on every PR. It installs dependencies, runs tests, and runs Snyk. If any step fails, the whole job fails and nothing merges.

The `auto-merge` job only runs when two conditions are both true: the PR was opened by `renovate[bot]` AND the `ci` job passed. It uses `gh pr merge --squash` which creates a real squash commit on main — this is what triggers `cd.yml`.

> **Why use `gh pr merge` instead of a third-party action?**  
> `gh pr merge` is GitHub's own CLI. The merge creates a real commit event on main which correctly fires the `push` event that `cd.yml` listens on. Third-party merge actions can use the API in ways that bypass this event.

## Step 5 — CD Workflow

Create `.github/workflows/cd.yml`:

```yaml
name: CD — Deploy to Vercel

on:
  push:
    branches:
      - main

  workflow_dispatch:

jobs:
  deploy:
    name: Deploy to Vercel Production
    runs-on: ubuntu-latest

    permissions:
      contents: read

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - name: Install Dependencies
        run: npm ci

      - name: Log Deployment Context
        run: |
          echo "========================================"
          echo "  Deploying to Vercel Production"
          echo "  Commit SHA : ${{ github.sha }}"
          echo "  Commit Msg : ${{ github.event.head_commit.message }}"
          echo "  Author     : ${{ github.event.head_commit.author.name }}"
          echo "  Triggered  : ${{ github.actor }}"
          echo "  Branch     : ${{ github.ref_name }}"
          echo "  Timestamp  : $(date -u '+%Y-%m-%dT%H:%M:%SZ')"
          echo "========================================"

      - name: Install Vercel CLI
        run: npm install --global vercel@latest

      - name: Pull Vercel Environment Information
        run: |
          vercel pull --yes \
            --environment=production \
            --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}

      - name: Deploy to Vercel Production
        id: vercel-deploy
        run: |
          DEPLOY_URL=$(vercel deploy --prod --yes \
            --token=${{ secrets.VERCEL_TOKEN }} 2>&1 | tail -1)
          echo "deploy_url=$DEPLOY_URL" >> $GITHUB_OUTPUT
          echo "Deployment URL: $DEPLOY_URL"
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}

      - name: Log Deployment Success
        run: |
          echo "========================================"
          echo "  Deployment Successful"
          echo "  URL: ${{ steps.vercel-deploy.outputs.deploy_url }}"
          echo "  Commit: ${{ github.sha }}"
          echo "  Time: $(date -u '+%Y-%m-%dT%H:%M:%SZ')"
          echo "========================================"
```

> **Key difference from CI:** CD only runs on `push` to `main`. It never runs on PRs. This separation means a failing PR can never accidentally deploy broken code.

## How Each File Works

### `renovate.json`

Renovate reads this file to understand your preferences. The most important settings:

- `schedule` — when Renovate runs
- `packageRules` — what to do with different types of updates (auto-merge patch, flag major)
- `dependencyDashboard` — creates a GitHub Issue as a central control panel for all pending updates
- `vulnerabilityAlerts` — enables security-specific PRs when Snyk Advisory database flags a package

### `ci.yml`

Triggered by: any Pull Request targeting `main`

Two jobs run in sequence:
1. `ci` — installs, tests, scans. Must pass before anything else runs.
2. `auto-merge` — only runs if the PR author is `renovate[bot]` AND `ci` passed.

The `needs: ci` line is critical — it creates a dependency between jobs so `auto-merge` never runs if CI failed.

### `cd.yml`

Triggered by: any push to `main` (including squash merges from auto-merge)

One job: deploys to Vercel. Logs commit SHA, author, timestamp, and the final production URL. The `workflow_dispatch` trigger lets you re-run a deploy manually from the Actions tab if needed.

### `vercel.json`

Tells Vercel how to build and route your app. `@vercel/node` handles Node.js apps. The route `(.*)` catches all paths and sends them to `index.js`.

## Troubleshooting

**`Dependencies lock file is not found`**

You forgot to commit `package-lock.json`. Run `npm install` locally and commit the file.

```bash
npm install
git add package-lock.json
git commit -m "add package-lock.json"
git push origin main
```

**`Invalid configuration option: logLevel` in Renovate**

Remove `"logLevel"` from `renovate.json`. It is not a valid config option for the hosted plan.

**`Post-upgrade task did not match any on allowedCommands list`**

Remove `postUpgradeTasks` from `renovate.json`. That feature requires a self-hosted Renovate instance. The free hosted plan does not support it.

**PRs not appearing in Pull Requests tab**

Check in order:
1. Is Renovate installed on your repo? Go to `github.com/apps/renovate` → Configure → verify your repo is listed.
2. Did you merge the "Configure Renovate" onboarding PR? Renovate will not create dependency PRs until that PR is merged.
3. Run `npm outdated` locally. If it returns nothing, all your packages are already up to date — Renovate has nothing to do.
4. Go to `developer.mend.io` and click the refresh icon to force a run.

**`failed to push some refs` when pushing locally**

Renovate pushed commits to your remote. Pull first, then push.

```bash
git pull origin main --rebase
git push origin main
```

**CI passing but PR not auto-merging**

The `auto-merge` job only runs for PRs opened by `renovate[bot]`. For your own PRs, you merge manually. Check the PR author in the Pull Requests tab.

**Vercel deploy failing**

Make sure all three secrets are set correctly: `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`. Re-run `vercel link` locally to regenerate `.vercel/project.json` and copy the values again.

## Key Concepts Summary

| Concept | What it means |
|---|---|
| **CI (Continuous Integration)** | Automatically test every change before it merges |
| **CD (Continuous Deployment)** | Automatically deploy every change that passes CI |
| **DevSecOps** | Security is built into the pipeline, not added at the end |
| **Security gate** | A pipeline step that blocks deployment if a security check fails |
| **Auto-merge** | A PR merges itself when all checks pass — no human needed |
| **Squash merge** | Combines all commits in a PR into one clean commit on main |
| **Artifact** | A file saved by a CI run (e.g. Snyk scan results) for later download |
| **GitHub Secrets** | Encrypted variables injected into workflows at runtime |
| **pull_request event** | Fires when a PR is opened, updated, or reopened — triggers CI |
| **push event** | Fires when a commit lands on a branch — triggers CD |

## What Happens When Snyk Finds a Vulnerability

This is the security gate in action. Here is the exact sequence:

1. Renovate opens a PR: `update express 4.17.0 → 5.2.1`
2. CI starts. `npm ci` installs `express 5.2.1` into `node_modules/`
3. Snyk scans `node_modules/express` specifically
4. Snyk finds a known CVE with medium or higher severity
5. The `Fail if Medium+ Vulnerabilities Found` step exits with code 1
6. The CI job fails — red ❌ in the Actions tab
7. The `auto-merge` job is skipped because `ci` failed
8. The PR stays open with a failed status
9. Nothing deploys to Vercel
10. You receive a notification and can investigate the vulnerability

This is DevSecOps working correctly. The pipeline protected production from a vulnerable dependency.
